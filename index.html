高并发抢购杂谈之三

Veröffentlicht am 2017-01-05

消息队列

我们知道对数据库进行更新操作(update,insert)是比较慢的，这个常常会成为我们在高并发场景下的瓶颈。这时候我们就会想这个可不可以异步去处理，比如说我们在内存里把订单生成好了就可以告诉用户购买成功了，然后我们再异步的去更新数据库。这时候我们就可以利用消息队列，我们在接受到请求，然后生成订单对象，之后通过消息队列发送出去，这时候就可以返回购买成功的消息了。最后再由订阅端来持久化到数据库。

Redis

如果我们异步的操作数据库，这时候我们会遇到一个问题就是，我们数据库里面的库存就不是我们的真实库存了，所以我们要把库存先缓存起来，然后直接扣减缓存里面的库存，这时候就可以用到Redis，我们只要把我们的库存保存到redis然后再利用乐观锁来减库存就可以了。

高并发抢购杂谈之二

Veröffentlicht am 2017-01-04

分表

在抢购的时候我们会面临大量的订单，这时候我们要对数据库进行大量的更新操作，这时候我们的数据库往往会成为我们系统的瓶颈，这时候我们可以对数据库进行分库分表来提高并发的更新速度。

Atlas

Atlas是奇虎360开源的一个MySQL代理中间件，我们可以使用Atlas来实现分表，这样我们可以不用改动我们的代码，便于维护，但是性能相对于直连数据库肯定会有所下降。
github地址:Atlas
安装和配置大家可以参考官方文档，下面我们就说说，订单表应该怎么分表。
假设我们订单主要涉及这三张表：
用户表：user
订单表：order
订单子表：order_line
我们通过Atlas执行操作时，Atlas会根据我们的分表字段来定位到相应的表，所以我们要先确定好我们的分表字段。
我们的客户端一般都是通过user_id来查询我们的订单，通过order_number来查询我们的订单详情。所以我们可以把order表的分表字段定为user_id，order_line表的分表字段定为order_number。

高并发抢购杂谈之一

Veröffentlicht am 2017-01-01

秒杀或者抢购的超卖问题：

假设50个人同时抢购1件商品，我们先查询商品是否有库存
select stock_quantity from sec_product WHERE id=?;
然后判断是否大于0，再减库存和插入订单
UPDATE sec_product SET stock_quantity=stock_quantity-1 WHERE id=?;
这时候就会出现问题，如果50个请求几乎同时到达，大家查询到的库存数量都是一样的，这时大家都会去减库存和生成订单，这是就是出现剩余库存为-49件，即超卖了49件的情况。
为了避免这种情况的出现，我们一般会把我们的减库存语句改成这样：
UPDATE sec_product SET stock_quantity=stock_quantity-1 WHERE id=? AND stock_quantity>0;
这时候我们的库存就不会减到负数了。
但是这样子还会出现一个问题，就是我们的update操作数据库是会加上锁的，获取到锁的用户去更新，而没有获取锁的用户也只能干等着，这样我们的用户得到反馈的速度可能会很慢，而且我们的服务器也可能因为资源耗尽而宕机（因为处理速度不如进来的速度快，会越积越多）。
这时候我们会再改进一下
先把库存查询出来
select stock_quantity from sec_product WHERE id=?;
然后在减库存的时候加上我们查询出来的库存数
UPDATE sec_product SET stock_quantity=stock_quantity-1 WHERE id=#{id} AND stock_quantity=?;
然后更新记录为0的话就直接返回没抢到，这样就不会阻塞了。
